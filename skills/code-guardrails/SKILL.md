# Code Guardrails

这是 PPW Codex 的轻量代码护栏 skill。目标是在真正有风险的代码修改前选择最小必要约束，避免范围蔓延、架构漂移、隐藏 MVP 取舍和无必要基础设施。

## 使用方式

先选择最小适用护栏，只应用相关小节，不要把它变成通用长 checklist。

- Tiny edits：不额外增加流程；如果触及数据、鉴权、prompt、公开 API 或架构边界，升级到对应护栏。
- Setup/tooling：使用「Setup 边界」；如果脚手架或多文件接线，补充「架构形状」。
- Feature delivery：使用「架构形状」「功能范围」「取舍披露」。
- Bug fix：使用「Bug fix」；如无法快速定位，回到 `AGENTS.md` 的系统排障规则。
- Refactor/cleanup：使用「架构形状」「重构边界」。
- High-risk ambiguity：编码前先澄清；交付时说明取舍。

## 架构形状

- 添加功能前先确认现有架构能否干净承载；不要因为某个文件离得近就继续塞职责。
- 如果新增责任会扩大混合用途文件、重复编排、模糊所有权或增加耦合，先抽出最小模块、组件、hook、service、schema 或 test fixture。
- 避免单文件同时承载 UI 渲染、数据访问、业务规则、校验、prompt 和副作用。
- route/page/handler 保持为编排层；可复用逻辑放到 feature、service、domain 或 application 模块。
- 大条件分支、重复状态接线、深层 callback、长 helper 串，是抽出命名单元的信号。

## Setup 边界

- 不引入新框架、包管理器、服务、插件、缓存或运行时，除非现有技术栈无法完成当前任务。
- 生成脚手架只保留当前阶段会实际使用的文件。
- 不把模板里的可选目录、示例配置、演示代码默认带进项目。

## 功能范围

- 只实现用户要求的主路径；不追加推测性的模式、设置、集成、后台入口或管理能力。
- 数据结构、API surface、UI 行为、字段和权限边界保持当前需求允许的最小形态。
- 需要扩展范围时，在继续编码前说明为什么当前需求无法闭合。

## Bug fix

- 有明显测试路径或核心逻辑受影响时，先复现或取得失败证据。
- 修最小根因，避免顺手扩大校验、权限、数据结构或行为语义。
- 复发概率高的失败模式要补回归测试。

## 重构边界

- 默认行为保持不变，除非用户明确要求行为变化。
- 不把重构和无关格式化、依赖升级、功能开发混在一起。
- 如果重构是为功能铺路，只做当前功能需要的最小结构调整。

## 取舍披露

快速交付可以接受，隐藏取舍不接受。

- 如果牺牲了安全、隐私、性能、扩展性、可靠性、可观测性、测试深度、可访问性、UX 细节或长期可维护性，最终回复必须说明。
- 说明优化目标、推迟或削弱了什么、为什么，以及下一步最小改进。
- 没有实质取舍时，不需要写空洞的取舍小节。

## 先澄清

以下情况不要静默假设：

- 可能暴露、导出或覆盖敏感数据。
- 改变 auth、payment、prompt safety 或公开 API 行为。
- 涉及数据迁移、删除、覆盖、回填。
- 会锁定重大架构、基础设施或供应商选择。
- 两种合理解释会导致明显不同实现。
